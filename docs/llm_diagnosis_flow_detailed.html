Detailed LLM Diagnosis Flow (Stage-by-Stage)

This guide explains each stage in your exact pipeline flow in simple but detailed review-ready terms.

Pipeline Flow (Reference)

[User symptom input] | v [Controller initializes state] | v [Ambiguity detection] | +--> Missing info? ---- Yes --> [Ask clarification question] --> [User answer] --> (loop back) | (until enough info / max turns) | No v [Hybrid RAG retrieval] (JSON + PDF chunks) | v [Relevance filtering (CRAG-style)] | v [DiagnosisGenerator builds prompt] - context chunks - conversation history - strict output format | v [LLM generates diagnosis text] (candidate comparison + final disease + treatment + prevention) | v [Parser/normalizer in code] - extract DISEASE - normalize aliases - extract sections | v [HallucinationChecker verification] - regenerate N times - compute agreement score | +--> Low agreement? ---- Yes --> [Safety gate: ask 1 more targeted question] --> (loop back) | No v [Final output to UI]

1) User symptom input

What happens User writes initial symptoms in the web app.
  • Input can be short ("yellow leaves") or detailed.
  • This is raw, human language and may be incomplete.

Why this stage matters: it is the base signal for everything after this.

2) Controller initializes state

Filediagnosis/controller.py
  • Creates a new conversation session in StateManager.
  • Stores initial query, resets previous turns, starts with low confidence.
  • Checks whether the input is relevant to coffee plant diagnosis.

3) Ambiguity detection

Filerag/ambiguity_detector.py
  • Scans input for symptom attributes: color, pattern, location, spread, timing.
  • Finds which attributes are missing.

If missing info exists: system asks follow-up question instead of early diagnosis.

4) Clarification loop (if needed)

Filesrag/clarification_gen.py, diagnosis/controller.py
  • Generates one focused question based on missing details.
  • User answer is added to state.
  • Confidence increases per useful answer.
  • Loop repeats until enough details or max turns reached.

5) Hybrid RAG retrieval

Filescore/retriever.py, core/json_retriever.py, core/vector_store.py
  • Retrieves evidence from two sources:
    • Structured JSON disease knowledge
    • Unstructured PDF document chunks
  • Uses multi-query variants (base query + history + keywords).
  • Merges candidates into a hybrid result set.

6) Relevance filtering (CRAG-style)

Filerag/retrieval_evaluator.py
  • Scores chunk relevance to the query.
  • Removes low-relevance chunks below threshold.
  • Deduplicates near-same chunks.
  • Keeps source diversity (not all chunks from one source).

7) Diagnosis prompt construction

Filediagnosis/diagnosis_generator.py
  • Builds final prompt with:
    • retrieved context chunks,
    • user history,
    • strict output format rules.
  • Asks for candidate comparison + final diagnosis + treatment + prevention.

Key idea: model is guided to produce structured medical-style output, not free chat.

8) LLM diagnosis generation (actual model step)

Filecore/llm_client.py
  • Prompt is sent to configured provider/model.
  • Model predicts text token-by-token based on prompt + retrieved evidence.
  • Returns candidate diseases, selected disease, reason, treatment, prevention text.

Important: this output can still be noisy, so post-processing is required.

9) Parser + normalizer in code

Filediagnosis/diagnosis_generator.py
  • Extracts fields from raw model text (DISEASE, REASON, TREATMENT, PREVENTION).
  • Normalizes disease aliases to canonical names.
  • Applies fallbacks if text is incomplete/short.

10) Hallucination verification

Filediagnosis/hallucination_checker.py
  • Re-generates diagnosis multiple times (N generations).
  • Extracts disease name from each generation.
  • Computes agreement score (majority match ratio).
  • Marks safe_to_finalize true/false based on threshold.

11) Safety gate decision

Filediagnosis/controller.py (_generate_final_diagnosis)
  • If verification is strong -> finalize diagnosis.
  • If weak and question budget remains -> ask one more targeted clarification question.
  • Then run generation/verification again with updated info.

12) Final output to UI

Fileui/streamlit_app.py
  • Shows disease, confidence, reason, treatment, prevention.
  • Displays verification status (consistent/warning, agreement score).
  • Shows top evidence snippets and source tags.
